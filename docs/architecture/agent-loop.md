# Agent Loop 架构说明

## 1. 目标

`agent loop` 是整个系统中最稳定的核心。后续不论增加工具、技能、任务、subagent 还是上下文压缩，都应尽量作为外部能力接入，而不是反复重写循环本身。

## 2. 最小循环

```text
用户输入
  -> 组装运行上下文
  -> 调用 provider
  -> 若返回普通文本，则结束
  -> 若返回工具调用，则执行工具
  -> 将工具结果回填到消息历史
  -> 再次调用 provider
  -> 直到没有新的工具调用
```

当前实现状态：

- 已支持最小的“工具调用 -> 工具执行 -> 工具结果回填 -> 再次请求 provider”闭环。
- 当前仍只支持串行处理工具调用，不支持并行执行、重试和更复杂的停止策略。
- 当前通过 `Message::ToolResult` 把工具输出回填到会话历史，供下一轮 provider 响应使用。
- 当前在工具失败时已经返回显式错误，不再通过 panic 中断流程。

当前已经落地的最小实现已覆盖两条基础分支：

- provider 返回纯文本后结束本轮；
- provider 返回工具调用后，执行工具、回填工具结果并重入 provider。

## 3. 核心对象

- `Message`
  - 表示系统、用户、助手、工具结果等消息。
- `AssistantBlock`
  - 表示助手返回的文本块或工具调用块。
- `ToolCall`
  - 表示模型发起的工具调用请求。
- `ToolResult`
  - 表示工具返回结果。
- `SessionState`
  - 表示一个可恢复会话的完整状态。
- `TurnContext`
  - 表示当前回合的运行上下文，如工作目录、技能、计划、配置。

## 4. 依赖关系

`agent loop` 只依赖抽象，不直接依赖具体实现：

- 依赖 `ChatProvider` trait，而不是具体 OpenAI 客户端。
- 依赖 `ToolRegistry` 抽象，而不是具体 `shell` 或 `read_file` 实现。
- 依赖 `SessionStore` 或 runtime 接口，而不是具体磁盘格式。

## 5. 停止条件

首版建议支持以下停止条件：

- provider 返回纯文本且不再要求工具调用。
- provider 返回的本轮响应中不再包含工具调用块。
- 超过最大工具调用轮次。
- 工具执行出错且策略要求终止。
- 用户主动中断。

## 6. 首版不做的复杂逻辑

- 并行工具调用
- 多 provider 回退
- 复杂重试策略
- context compaction
- subagent 调度

这些能力后续都通过 loop 周边模块扩展，而不是推翻 loop 本身。

## 7. 当前未闭环项

- `chat` 命令尚未把 system prompt、provider 与 `run_agent_loop` 完整串接。
- 当前 loop 仍直接知道最小内置工具集合，后续应逐步收敛到更清晰的注册表注入边界。
- 当前测试重点覆盖成功路径与单个失败路径，多工具连续调用和更多边界情况仍待补充。
