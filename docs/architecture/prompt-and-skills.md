# Prompt 与 Skills 架构说明

## 1. 目标

提示词系统必须可组合、可维护、可替换。不能把所有规则直接硬编码成一个无法管理的大字符串。

## 2. 组装顺序

建议按以下顺序组装：

1. 基础系统提示
2. 仓库级规则
3. 当前任务上下文
4. 技能内容
5. 运行时边界与安全约束

这样做的好处是来源清晰，也便于后续替换其中某一层。

## 3. 基础系统提示

负责表达 agent 的基础身份与通用行为，例如：

- 它是一个本地 coding agent
- 它应优先调用工具而不是臆测
- 它应尽量保留可恢复状态

## 4. 仓库级规则

来自项目根目录的 `AGENTS.md`，用于表达：

- 当前阶段路线
- 禁止事项
- 推荐阅读顺序
- 开发约束

## 5. 当前任务上下文

来自本轮用户输入、当前会话摘要、当前计划状态。

## 6. 技能内容

技能优先从本地 `skills/` 目录读取，约定格式为：

```text
skills/
└─ <skill-name>/
   └─ SKILL.md
```

首版只做本地文件加载，不做远程下载与安装市场。

## 7. 首版建议能力

- 显式传入 `--skill <name>` 加载技能
- 或通过任务配置自动附加少量技能
- 把技能内容作为独立段落插入系统上下文

当前最小实现约定：

- `SkillLoader` 从 `skills/<name>/SKILL.md` 读取本地技能内容
- `chat --skill <name>` 负责触发本地技能加载入口
- 基础提示按固定顺序组装：基础系统提示、仓库级规则、当前任务上下文、技能内容、运行时边界
- `update_plan` 作为最小内置工具提供基础计划文本更新能力

当前还未完全闭环的部分：

- `chat --skill` 已能触发本地技能加载并构造 prompt 文本，但尚未把该 prompt 接到真实 provider 执行链路。
- `update_plan` 目前是最小文本化工具，而不是真正的计划状态存储接口。
- CLI 层针对 `--skill` 的直接行为测试仍可继续补充。

## 8. 首版边界

首版不做：

- 技能依赖解析
- 技能自动安装
- 多层技能继承
- 技能版本管理

先确保“能加载、能使用、可追踪来源”。
